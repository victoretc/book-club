## Блок 1: Архитектура и работа с данными

### 1.1. Прямые вызовы API из компонентов
* **Наблюдение:** Компоненты (`ClubForm.vue`) напрямую используют `axios` для API-запросов, минуя сторы Pinia.
  ```typescript
  // ClubForm.vue, onMounted
  const response = await axios.get(`/api/v1/clubs/${props.clubId}/`)

  // ClubForm.vue, onSubmit
  await axios.post('/api/v1/clubs/', values)
  ```
* **Последствия:**
  * **Дублирование логики:** Логика взаимодействия с API разбросана по приложению (часть в сторах, часть в компонентах).
  * **Децентрализация состояния:** Обновления состояния не проходят через единый источник (стор), что усложняет отладку.
  * **Нарушение принципа единой ответственности:** Компоненты представления берут на себя ответственность за получение и отправку данных.
* **Рекомендация:** Перенести всю логику API-запросов в соответствующие сторы Pinia. Компоненты должны вызывать экшены из сторов, а не `axios` напрямую.

### 1.2. Неэффективное обновление состояния
* **Наблюдение:** После действий, изменяющих данные (`joinClub`, `leaveClub`), происходит полная перезагрузка всего списка клубов с сервера.
  ```typescript
  // stores/clubs.ts
  async joinClub(clubId: number) {
    // ...
    await this.fetchClubs(this.pagination.currentPage, this.pagination.pageSize) // Перезагрузка всего списка
  }
  ```
* **Последствия:** Создается избыточная нагрузка на сеть и сервер, UX ухудшается из-за "моргания" интерфейса.
* **Рекомендация:** Вместо полной перезагрузки обновлять состояние локально (оптимистично или после подтверждения от сервера). Например, обновить только один объект клуба в массиве `this.clubs`.

### 1.3. Логика авторизации в компоненте
* **Наблюдение:** Логика проверки прав доступа (является ли пользователь владельцем клуба) находится в хуке `onMounted` компонента `ClubForm.vue`.
  ```typescript
  // ClubForm.vue, onMounted
  const authStore = useAuthStore()
  if (authStore.user && Number(response.data.owner?.id) !== Number(authStore.user.id)) {
    router.push({ name: 'clubs' })
  }
  ```
* **Последствия:** Смешение логики представления с бизнес-логикой и правилами доступа. Проверку легко забыть в других местах, где требуются те же права.
* **Рекомендация:** Перенести логику проверки прав доступа в навигационные хуки Vue Router (`beforeEnter` для маршрута).

### 1.4. Бизнес-логика в компонентах-представлениях (Views)
* **Наблюдение:** Компоненты-представления (`src/views`) содержат бизнес-логику вместо того, чтобы просто компоновать UI-компоненты.
* **Последствия:**
  * **Нарушение принципа единой ответственности:** Роль "view" — действовать как страница, собирая и отображая UI-компоненты. Она не должна заниматься обработкой данных или управлением состоянием.
  * **Низкая переиспользуемость:** Логику, "зашитую" в view, нельзя легко переиспользовать.
* **Рекомендация:** Компоненты в `src/views` должны быть максимально простыми. Их задача — импортировать компоненты из `src/components`, располагать их на странице и передавать им данные. Вся интерактивная логика должна быть инкапсулирована в дочерних компонентах.

## Блок 2: Обработка ошибок и UX

### 2.1. Отсутствие обратной связи для пользователя при ошибках API
* **Наблюдение:** Ошибки API в основном логируются в консоль, пользователь не получает информации о проблеме.
  ```typescript
  // ClubForm.vue, onSubmit
  } catch (error) {
    if (axios.isAxiosError(error) && error.response?.data) {
      // Ошибки валидации от сервера просто логируются
      console.error('Server validation errors:', error.response.data)
    }
    // Пользователь ничего не видит
  }
  ```
* **Последствия:** Пользователь не понимает, почему его действие не удалось, что приводит к плохому UX.
* **Рекомендация:**
  * Создать централизованный обработчик ошибок.
  * Отображать пользователю понятные сообщения об ошибках (toast-уведомления, ошибки под полями формы).

### 2.2. Неинформативные сообщения об ошибках
* **Наблюдение:** При ошибке загрузки данных в сторе устанавливается статичное, заранее заданное сообщение.
  ```typescript
  // stores/clubs.ts
  } catch (error) {
    this.error = 'Не удалось загрузить список клубов' // Всегда одно и то же сообщение
    console.error('Error fetching clubs:', error)
  }
  ```
* **Последствия:** Теряется контекст ошибки (403, 500, и т.д.).
* **Рекомендация:** Анализировать код ошибки (`error.response.status`) и ее тело для предоставления релевантной информации.

## Блок 3: Стиль кода и лучшие практики

### 3.1. Порядок блоков в `<script setup>`
* **Наблюдение:** Внутри `<script setup>` блоки кода расположены неконсистентно.
* **Последствия:** Отсутствие единого порядка усложняет чтение и навигацию по коду компонента.
* **Рекомендация:** Придерживаться общепринятого соглашения по порядку блоков:
  1. `import`
  2. `defineProps`, `defineEmits`
  3. `use...()` (Composable функции)
  4. `ref`, `reactive`, `computed`
  5. Методы и функции-обработчики
  6. `watch`, `watchEffect`
  7. Хуки жизненного цикла (`onMounted`, etc.)

### 3.2. Встроенная логика в хуках жизненного цикла
* **Наблюдение:** Логика инициализации описывается прямо внутри анонимной функции хука.
  ```vue
  // src/views/UserProfile.vue
  onMounted(() => {
    if (authStore.user) {
      editForm.value = { /* ... */ }
    }
  })
  ```
* **Последствия:**
  * **Читаемость:** Хуки становятся перегруженными.
  * **Повторное использование:** Логику нельзя переиспользовать без дублирования.
* **Рекомендация:** Выносить логику из хуков в отдельные именованные функции. Хук должен только вызывать эту функцию.
  ```vue
  function initializeFormData() {
    if (!authStore.user) return;
    editForm.value = { /* ... */ }
  }

  onMounted(initializeFormData)
  ```

### 3.3. Стиль обработки асинхронных операций
* **Вопрос:** Стиль `async/await` vs `.then/.catch`.

### 3.4. Ручное формирование параметров запроса в `axios`
* **Наблюдение:** Параметры GET-запроса формируются вручную с помощью `URLSearchParams`.
  ```typescript
  // stores/clubs.ts
  const searchParams = new URLSearchParams(/*...*/)
  const response = await axios.get<PaginatedClubList>(
    `/api/v1/clubs/?${searchParams.toString()}`,
  )
  ```
* **Последствия:** Избыточный код, игнорирование встроенных возможностей `axios`.
* **Рекомендация:** Использовать встроенную в `axios` обработку параметров, передавая объект в свойстве `params`.
  ```typescript
  // stores/clubs.ts
  const response = await axios.get<PaginatedClubList>(`/api/v1/clubs/`, {
    params,
  })
  ```

## Блок 4: Безопасность

### 4.1. Валидация ссылок на стороне клиента
* **Наблюдение:** Валидация ссылки на Telegram (`.matches(/^https:\/\/t.me\//)`) присутствует только на стороне клиента.
* **Последствия:** Клиентская валидация может быть обойдена, что создает потенциальную уязвимость.
* **Рекомендация:** Всегда дублировать валидацию на стороне сервера. При отображении внешних ссылок добавлять `rel="noopener noreferrer"` к тегу `<a>`.
